# User Stories: english-coach v1 — voice-first тренажёр

**Epic:** english-coach v1 — заговорить на английском через голосовые сессии с AI-коучем
**Источник:** [PRD.md](PRD.md) (scope) · [IMPLEMENTATION_PLAN.md](IMPLEMENTATION_PLAN.md) (инженерные решения)
**Date:** 2026-07-07 · **Status:** Ready for execution

**Калибровка оценок.** Команда — один ученик + AI-агенты, поэтому Fibonacci-поинты меряют относительную сложность, а не человеко-дни: 1 pt ≈ одна короткая агент-сессия (~30–45 мин), 5 pt ≈ большая сессия с живым тестированием. Паттерн разбивки — гибрид: по шагам пути пользователя (установка → знакомство → ежедневный разговор → неделя под коучем → прогресс) с нарастанием сложности внутри ядра.

## Стадия 1 — Установка и доверие (Фаза 0)

### US-001: Гарантированное сохранение результатов сессии

**Role:** As ученик
**Action:** I want чтобы результаты каждой сессии записывались в базы независимо от способа вызова механизма персиста
**Value:** so that прогресс и ошибки не терялись молча — память коуча и есть продукт
**Priority:** P0
**Story Points:** 1

**Acceptance Criteria:**
- [ ] Given валидный JSON-отчёт сессии, When он передан аргументом командной строки, Then все 6 баз обновлены идентично варианту передачи через stdin
- [ ] Given некорректный JSON-отчёт, When персист запускается, Then базы не изменены, возвращена понятная ошибка с ненулевым кодом выхода
- [ ] Given персист стартует, When начинается запись, Then до записи создан снапшот баз и данные восстановимы при сбое

**Dependencies:** none
**Tech notes:** фикс из upstream PR #11 (3 строки argv-ветки), Anki-экспорт и import-lr из того PR не берём; тест-кейс на argv-путь в `tests/test_update_db.py`.

### US-002: Личные данные не попадают в публичный форк

**Role:** As ученик
**Action:** I want чтобы профиль, ошибки, логи и транскрипты физически не могли попасть в git публичного репозитория
**Value:** so that пользоваться инструментом без самоцензуры
**Priority:** P0
**Story Points:** 1

**Acceptance Criteria:**
- [ ] Given проведена сессия, создавшая профиль/ошибки/транскрипт, When выполняется `git status` в форке, Then личные файлы не отображаются (только заглушки и примеры)
- [ ] Given появился новый тип личного файла, видимый git, When готовится коммит, Then коммит останавливается до добавления файла в исключения

**Dependencies:** none
**Tech notes:** проверить и дополнить `.gitignore` (`data/*.json`, `results/*.md`, бэкапы); `.gitkeep` остаются.

### US-003: Форк идентифицирован и дёшево синхронизируется с upstream

**Role:** As мейнтейнер
**Action:** I want чтобы форк имел собственную идентичность и настроенный канал обновлений из upstream
**Value:** so that получать улучшения fluent черри-пиком, не теряя свои доработки
**Priority:** P1
**Story Points:** 1

**Acceptance Criteria:**
- [ ] Given настроен remote upstream, When выполняется fetch, Then история upstream доступна и коммиты можно черри-пикать
- [ ] Given обновлена идентичность (имя english-coach, версия, ссылки, README-шапка), When новая сессия запускается из папки, Then хуки и скиллы видны (SessionStart печатает статистику)
- [ ] Given черри-пик upstream-коммита конфликтует с нашими правками, When идёт синк, Then конфликт ограничен секциями примеров в скиллах и не затрагивает новые файлы coach/voice-слоя

**Dependencies:** none
**Tech notes:** режим работы — папка-проект, не plugin install; версия `0.3.0-ec.1`.

## Стадия 2 — Знакомство (Фаза 1)

### US-004: Профиль ученика EN/RU с голосовыми предпочтениями

**Role:** As ученик
**Action:** I want один раз рассказать коучу о себе — языки, цели, голосовые предпочтения
**Value:** so that каждая сессия начиналась с моего контекста, а не с чистого листа
**Priority:** P0
**Story Points:** 2

**Acceptance Criteria:**
- [ ] Given чистая установка без профиля, When онбординг пройден, Then профиль содержит: target English, native Russian, мотивация work (созвоны, питчи, small talk, мысль без русского костыля), уровень «определить разговором», дневная цель минут, блок voice (озвучка вкл, голос, темп, способ ввода)
- [ ] Given онбординг прерван на середине, When ученик возвращается, Then продолжение с места остановки, заполненные поля сохранены
- [ ] Given профиль без зафиксированного уровня, When завершена первая живая сессия, Then уровень (или диапазон) записан в профиль без формального теста

**Dependencies:** none
**Tech notes:** штатный `/fluent-setup`; блок `voice` — своё поле в `preferences`, fluent его не трогает.

### US-005: Коуч заранее знает типичные ловушки русскоязычных

**Role:** As ученик
**Action:** I want чтобы коуч с первой сессии следил за типичными ошибками русскоязычных
**Value:** so that прицельный фидбек начался сразу, без недель «холодного старта»
**Priority:** P0
**Story Points:** 1

**Acceptance Criteria:**
- [ ] Given банк ошибок без личной истории, When сид выполнен, Then в банке 7 паттернов-«вахтёров» с frequency 0 и пометкой «typical RU-speaker pattern — watch»: артикли, Present Perfect vs Past Simple, предлоги, порядок слов в вопросах, will после if, кальки, countable/uncountable
- [ ] Given проведено 3 сессии без ошибок конкретного паттерна, When формируется разбор, Then frequency паттерна остаётся 0 и паттерн не попадает в фокусы

**Dependencies:** none (параллельно с US-004)
**Tech notes:** frequency растёт только от реальных ошибок — не выдумывать.

### US-006: Контент упражнений — английский

**Role:** As ученик
**Action:** I want чтобы примеры и разговорные филлеры в упражнениях были английскими
**Value:** so that тренировать именно английскую речь, а не переводить голландские заготовки
**Priority:** P0
**Story Points:** 2

**Acceptance Criteria:**
- [ ] Given скиллы говорения, обучения и лексики, When ученик проходит упражнение, Then все примеры и филлеры английские (well…, actually…, to be fair…, that makes sense), голландских не осталось
- [ ] Given diff к upstream после правок, When ревью изменений, Then затронуты только секции примеров существующих скиллов (принцип «не ломать upstream»)

**Dependencies:** none (параллельно с US-004, US-005)
**Tech notes:** объяснения — на английском по умолчанию, русский по запросу; зафиксировать в локальном контексте форка (CLAUDE.md, аддитивно).

## Стадия 3 — Ежедневный разговор (Фаза 2, ядро)

### US-007: Провести голосовой разговор по рабочему сценарию

**Role:** As ученик
**Action:** I want вести 10–15-минутный голосовой разговор с коучем по рабочему сценарию — говорю голосом, ответы озвучиваются
**Value:** so that тренировать спонтанную речь в ситуациях, которые ждут меня на работе
**Priority:** P0
**Story Points:** 5

**Acceptance Criteria:**
- [ ] Given профиль с включённой озвучкой и отсутствие плана на сегодня, When запускается `/talk`, Then предложены 3 сценария по слабым темам из библиотеки (созвон с инвестором, статус-митинг, small talk, объяснение продукта, спор, собеседование, travel/бытовые) и после выбора идёт разговор: реплики коуча 2–3 предложения, без markdown и эмодзи, каждая озвучивается сразу
- [ ] Given идёт разговор, When ученик делает ошибку, Then коуч отвечает как собеседник без исправлений — ноль фидбека до конца разговора
- [ ] Given распознавание выдало бессвязный фрагмент, When коуч получает ввод, Then переспрос как живой собеседник («Sorry, you mean…?») и фрагмент не учитывается как ошибка
- [ ] Given ученик застрял и молчит, When коуч это видит, Then подсказка фразы и разговор продолжается без превращения в урок
- [ ] Given озвучка выключена в профиле, When запускается `/talk`, Then сессия идёт текстом по тем же правилам
- [ ] Given разговор длится ~10–15 минут или произнесена стоп-фраза, When условие срабатывает, Then разговор завершается и передаётся в разбор

**Dependencies:** US-004 (голосовые предпочтения — Mockable to decouple: дефолты позволяют начать без профиля)
**Tech notes:** контекст перед разговором — уровень, слабые места, 2–3 due-фразы SRS вплести; вход — надиктованный текст (пунктуацию/капитализацию игнорировать); озвучка — Bash-вызов после каждой реплики.

### US-008: Получить разбор и пополнить память после разговора

**Role:** As ученик
**Action:** I want после разговора получать разбор — ошибки, полезные фразы, фокус — с записью в мою память
**Value:** so that каждый разговор делал следующий лучше, а не растворялся
**Priority:** P0
**Story Points:** 3

**Acceptance Criteria:**
- [ ] Given завершён разговор ≥10 минут, When формируется разбор, Then показаны ровно 4 блока: топ-3 реальные ошибки с объяснениями (только communication-blocking или повторяющиеся) → банк ошибок; 5 полезных фраз → SRS типом phrase; 1 фокус на следующую сессию; оценка communication-first → mastery; все базы обновлены, транскрипт сохранён
- [ ] Given разговор был активностью недельного плана, When персист завершён, Then активность помечена done со ссылкой на сессию (при отсутствии плана шаг молча пропускается)
- [ ] Given сессия прервана до разбора, When ученик возвращается, Then транскрипт сохранён с пометкой partial, базы не изменены, предложено завершить разбор или начать заново; сессия не идёт в зачёт метрики

**Dependencies:** US-007 (данные разговора), US-001 (надёжный персист)
**Tech notes:** персист штатным механизмом fluent (`command_used: "/talk"`, `skills_practiced: ["speaking"]`); транскрипт → `results/talk-session-{NNN}.md`.

### US-009: Первая живая сессия подтверждает v0

**Role:** As ученик
**Action:** I want провести полную живую сессию и откалибровать поток по её итогам
**Value:** so that объявить v0 готовым на основании реального использования, а не демо
**Priority:** P0
**Story Points:** 2

**Acceptance Criteria:**
- [ ] Given настроенный голосовой контур, When проведена полная сессия ≥10 минут со стоп-фразой, Then латентность хода ≤5 секунд, разбор корректен, базы обновлены, транскрипт на месте — v0 принят
- [ ] Given распознавание систематически искажает речь с акцентом, When идёт калибровка, Then выбрано и зафиксировано решение (модель распознавания крупнее или запасной способ ввода) и настройки обновлены
- [ ] Given темп или длина озвучиваемых реплик утомляют, When идёт калибровка, Then темп/длина скорректированы в профиле и повторная проверка пройдена

**Dependencies:** US-007, US-008
**Tech notes:** чек-пойнты калибровки — из плана §2.3 (распознавание, длина реплик, латентность, качество записей в базах).

> **🎤 Гейт после Стадии 3: 5 проведённых живых сессий.** Стадия 4 не начинается раньше — защита от «залипнуть в инженерку» (принцип №5 плана). Гейт — не стори: это использование продукта, ~1.5–2 недели календарно.

## Стадия 4 — Неделя под управлением коуча (Фаза 3)

### US-010: Получить план недели, подстроенный под мои данные

**Role:** As ученик
**Action:** I want получать план недели, построенный из моих слабых мест, очереди повторений и ретро прошлой недели
**Value:** so that не решать каждый день «чем заняться» и не бросить через две недели
**Priority:** P1
**Story Points:** 5

**Acceptance Criteria:**
- [ ] Given проведено ≥5 живых сессий и накоплены mastery/ошибки/SRS, When запускается `/coach-plan`, Then после короткого интервью (занятость, пожелания, что бесило) сгенерирован план: 3–4 talk, 1 listen+discuss с материалом по интересам и уровню, 1 write, 1–2 review; суммарная нагрузка ≤ дневная цель × доступные дни; план показан таблицей, правки внесены диалогом, итог записан
- [ ] Given первый запуск без ретро и с малым объёмом данных, When генерируется план, Then план строится от профиля и сид-паттернов и помечается калибровочной неделей
- [ ] Given прошлая неделя выполнена менее чем наполовину, When генерируется новый план, Then неделя облегчена, пропущенное перенесено или снято, причины спрошены в интервью
- [ ] Given очередь повторений пуста и ошибок мало, When генерируется план, Then talk-активностей больше, review сокращён до одного

**Dependencies:** US-008 (накопленные данные), гейт 5 сессий
**Tech notes:** схема `weekly-plan.json` — план §3.1 (`type ∈ {talk, listen, discuss, write, review, vocab}`, `status ∈ {planned, done, skipped}`, блок retro); каданс — вс ретро → пн план, синхронно с Chaush OS.

### US-011: Открыл терминал — сразу знаю, что делать сегодня

**Role:** As ученик
**Action:** I want видеть активность дня прямо в приветствии и запускать её одним подтверждением
**Value:** so that начинать занятие без силы воли и выбора
**Priority:** P1
**Story Points:** 3

**Acceptance Criteria:**
- [ ] Given план недели с активностью на сегодня, When открывается терминал, Then приветствие содержит одну строку с активностью дня; `/coach-today` предлагает её и по согласию запускает целевой скилл с параметрами из плана (сценарий, тема, материал)
- [ ] Given на сегодня write-активность, When `/coach-today` её запускает, Then существующий скилл письма получает рабочую тему из плана (письмо инвестору, апдейт команде, ответ на возражение)
- [ ] Given вчерашняя активность пропущена, When запускается `/coach-today`, Then нейтральное предложение нагнать или перепланировать — «пропуск — это данные, не провал», активность не помечается провалом
- [ ] Given плана недели нет, When запускается `/coach-today`, Then предложение составить план (если гейт пройден) или провести свободную talk-сессию
- [ ] Given пропущено несколько дней подряд, When запускается `/coach-today`, Then предложение пересобрать остаток недели вместо догонялок

**Dependencies:** US-010 (план как источник данных)
**Tech notes:** вывод в приветствии — аддитивное дополнение SessionStart-хука (в конец вывода); выполненную активность помечает done тот скилл, который её выполнил.

### US-012: Обсудить послушанный материал голосом

**Role:** As ученик
**Action:** I want обсуждать голосом подкаст/видео из плана — пересказ, мнение, спор с автором
**Value:** so that превращать пассивное слушание в активную речь
**Priority:** P1
**Story Points:** 3

**Acceptance Criteria:**
- [ ] Given listen-активность попадает в план, When план создаётся, Then к материалу выданы 3 вопроса-ориентира — на что обратить внимание при прослушивании
- [ ] Given материал прослушан, When запускается `/discuss`, Then голосовое обсуждение идёт по механике `/talk` (озвучка, ноль фидбека в потоке, отложенный разбор): пересказ, мнение, спор с тезисами; новая лексика из материала добавлена в SRS; разбор идентичен разбору talk; обе активности (listen и discuss) помечены done
- [ ] Given материал не прослушан к дню обсуждения, When запускается `/discuss`, Then discuss переносится и предлагается слот для прослушивания
- [ ] Given материал оказался слишком сложным, When идёт обсуждение, Then обсуждается понятая часть, а факт отмечается для калибровки подбора в следующем плане

**Dependencies:** US-007 (разговорная механика — интерфейсная зависимость), US-010 (связка listen→discuss в плане)
**Tech notes:** связь активностей — поле `followup`/`about` в схеме плана.

## Стадия 5 — Видеть прогресс (Фаза 4, опционально)

### US-013: Видеть минуты говорения и speaking-streak

**Role:** As ученик
**Action:** I want видеть минуты говорения в неделю и streak именно по речи
**Value:** so that понимать, растёт ли главное — практика, а не активность «вообще»
**Priority:** P2
**Story Points:** 2

**Acceptance Criteria:**
- [ ] Given накоплено ≥2 недель данных, When запрашивается прогресс, Then показаны минуты говорения по неделям (из сессий talk и discuss), speaking-streak и динамика топ-ошибок
- [ ] Given данных меньше недели, When запрашивается прогресс, Then показана только текущая неделя без трендов
- [ ] Given в логах есть прерванные сессии без длительности, When считаются минуты, Then partial-сессии исключены из суммы и показаны сноской

**Dependencies:** US-008 (источник данных)
**Tech notes:** только новое представление существующих баз, без новых источников данных.
