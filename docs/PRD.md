## 1. Background and Goals

### 1.1 Background

**Почему сейчас.** Ученику нужен разговорный английский для работы: созвоны, питчи, small talk, способность формулировать мысль без «русского костыля». Классические форматы (курсы, преподаватель по расписанию) не встраиваются в ритм ученика; существующие AI-чаты дают разговор, но не ведут по программе и не накапливают память об ошибках.

**Что будет, если не делать.** Практика говорения останется нерегулярной, прогресс — неизмеримым; типичные ошибки русскоязычных будут воспроизводиться годами, потому что их никто систематически не отслеживает.

**Что меняется, если сделать.** Появляется личный voice-first коуч в терминале: живой голосовой разговор по рабочим сценариям, автоматический разбор ошибок в персональную память (SM-2), недельная программа занятий с ретро. База — зрелый open-source тренажёр fluent (6 JSON-баз, SM-2, хуки), дорабатываем только голосовой и коуч-слои.

**Ключевой продуктовый принцип:** инструмент — средство, не цель. Метрика успеха — проведённые голосовые сессии, а не количество фич. Инженерка не съедает практику: после MVP голосового слоя — сразу живые сессии.

## 2. Requirement Overview

Voice-first тренажёр английского на базе форка fluent: живой голосовой разговор по рабочим сценариям с отложенным разбором ошибок, персональная память ученика (SM-2) и недельная программа занятий под управлением AI-коуча.

## 3. Detailed Feature Design

Декомпозиция: путь ученика (открыл терминал → узнал план дня → поговорил → получил разбор → память обновилась → неделя закрылась ретро) + CRUD по ключевым объектам данных (профиль, план недели, ошибки, SRS-фразы, транскрипты).

### 3.1 Голосовая разговорная сессия `/talk`

**Description:** Живой голосовой разговор с коучем 10–15 минут по заданному сценарию, без оценок в потоке, с разбором после завершения.
**User story:** As ученик, I want разговаривать с коучем голосом по рабочим сценариям, so that натренировать спонтанную речь для реальных созвонов.
**Priority:** P0 (must launch) — это ядро продукта; без него всё остальное не имеет смысла.

**Business rules:**
1. Перед разговором коуч загружает контекст ученика: уровень, слабые места, due-фразы из SRS-очереди; 2–3 due-фразы естественно вплетаются в разговор.
2. Выбор сценария: если в недельном плане есть talk-активность на сегодня — берётся её сценарий; иначе коуч предлагает 3 сценария на выбор, подбирая по слабым темам из библиотеки: созвон с инвестором, статус-митинг, small talk, объяснение продукта, спор/отстаивание позиции, собеседование, travel/бытовые ситуации.
3. Реплики коуча — 2–3 предложения, разговорный тон, без markdown и без эмодзи (текст озвучивается как есть); каждая реплика озвучивается сразу после вывода.
4. Ноль фидбека в потоке разговора: ошибки и удачные фразы накапливаются молча; любые оценки — только в разборе после завершения.
5. Вход ученика — надиктованный текст: пунктуация и капитализация игнорируются при оценке; явно бессвязный фрагмент распознавания → коуч переспрашивает как живой собеседник («Sorry, you mean…?») и не записывает это в ошибки.
6. Если ученик застрял — коуч подсказывает фразу и продолжает разговор, не превращая подсказку в урок.
7. Завершение разговора: по достижении ~10–15 минут или по стоп-фразе ученика (например, «let's wrap up»).
8. Разбор после разговора включает ровно четыре блока: (а) топ-3 ошибки с объяснениями — в mistakes-db попадают только реальные ошибки, блокирующие коммуникацию или повторяющиеся; (б) 5 полезных/натуральных фраз — в SRS-очередь типом phrase; (в) 1 фокус на следующую сессию; (г) оценка говорения по communication-first схеме fluent — в mastery.
9. Персист сессии: обновление всех баз через штатный механизм fluent, транскрипт сохраняется в `results/`; если сессия была активностью недельного плана — активность помечается `done` с ссылкой на сессию.

**Interaction flow:**
1. Ученик входит: команда `/talk` из терминала (или по подсказке `/coach-today`).
2. Коуч называет сценарий (из плана) или предлагает 3 на выбор → ученик выбирает голосом/текстом.
3. Разговор: реплика коуча → озвучка → ученик надиктовывает ответ → следующая реплика; цикл 10–15 минут.
4. Success: стоп-фраза или таймбокс → разбор (топ-3 ошибки, 5 фраз, фокус, оценка) → базы обновлены, транскрипт сохранён → короткое резюме и прощание.
5. Failure: см. таблицу исключений — сессия деградирует в текстовый режим или завершается с сохранением того, что успели.

**Exception handling:**
| Scenario | Handling | User-facing message |
|---|---|---|
| Распознавание выдало бессвязный текст | Переспросить как собеседник, фрагмент не учитывать в ошибках | «Sorry, you mean…?» |
| Озвучка недоступна или раздражает | Переключение `tts_enabled: false` → сессия продолжается текстом | «Switching to text mode — let's keep going.» |
| Латентность хода превышает целевые 5 секунд систематически | Коуч укорачивает реплики; факт фиксируется для калибровки | — (не прерывать поток) |
| Ученик молчит / застрял | Подсказать фразу, продолжить разговор | «You could say: …» |
| Сессия прервана до разбора (закрыт терминал, стоп без wrap-up) | Транскрипт сохраняется с пометкой partial; базы не обновляются; активность плана остаётся `planned`; сессия не идёт в зачёт метрики | При следующем входе: «Last talk was cut short — want to finish the review or start fresh?» |

### 3.2 Персонализация под русскоязычного ученика

**Description:** Профиль ученика (EN ← RU, цели, голосовые предпочтения) и предзаведённые типичные паттерны ошибок русскоязычных как «вахтёры» для наблюдения.
**User story:** As ученик, I want чтобы коуч с первой сессии знал мой язык, цели и типичные ловушки русскоязычных, so that фидбек был прицельным сразу, без недель «холодного старта».
**Priority:** P0.

**Business rules:**
1. Профиль заводится штатным онбордингом fluent: целевой язык English, родной Russian, мотивация — work (созвоны, питчи, small talk, формулировать мысль без русского костыля).
2. Текущий уровень не тестируется формально — определяется разговором в первой сессии и фиксируется в профиле.
3. Голосовые предпочтения хранятся в профиле в блоке `voice`: включена ли озвучка, выбранный голос, темп, способ ввода.
4. В банк ошибок предзаводятся паттерны-«вахтёры» с нулевой частотой и пометкой «typical RU-speaker pattern — watch»: артикли (a/the/∅), Present Perfect vs Past Simple, предлоги (depends **on**, listen **to**), порядок слов в вопросах, will после if, кальки («I feel myself», «very actual»), countable/uncountable (advices, informations).
5. Частота ошибок не выдумывается: счётчики растут только от реальных ошибок в сессиях.
6. Примеры и conversational fillers в скиллах говорения/обучения/лексики — английские (well…, actually…, to be fair…, that makes sense); объяснения — на английском по умолчанию, русский — по запросу ученика или для критически непонятых мест.

**Interaction flow:**
1. Ученик входит: онбординг-команда при первом запуске.
2. Диалог онбординга → профиль записан → сид-паттерны в банке ошибок.
3. Success: первая же `/talk`-сессия использует профиль и следит за паттернами.
4. Failure: онбординг прерван → при следующем входе предлагается продолжить с места остановки.

**Exception handling:**
| Scenario | Handling | User-facing message |
|---|---|---|
| Онбординг прерван на середине | Сохранить заполненные поля, при следующем запуске продолжить | «Let's pick up where we left off.» |
| Уровень не удалось уверенно определить в первой сессии | Записать диапазон, уточнять в следующих 2–3 сессиях | «I'll refine my estimate over the next sessions.» |
| Сид-паттерн не проявляется в речи ученика | Частота остаётся 0; паттерн не попадает в фокусы и разборы | — |

### 3.3 Приватность данных ученика

**Description:** Личные данные (профиль, ошибки, логи, транскрипты) никогда не попадают в публичный репозиторий форка; бэкап живёт в приватном контуре департамента.
**User story:** As ученик, I want чтобы моя учебная история физически не могла утечь в публичный форк, so that пользоваться инструментом без самоцензуры.
**Priority:** P0 — нарушение необратимо (форк публичный, приватным его сделать нельзя).

**Business rules:**
1. Все личные данные (`data/`, `results/`, бэкапы) исключены из git публичного форка; в репозитории остаются только заглушки-примеры.
2. Бэкап данных — локальные снапшоты штатного механизма fluent; долговременный бэкап — в приватном контуре (репо corp-english или отдельный приватный репо данных), не в форке.
3. Любой коммит в форк проходит проверку: статус git не должен показывать личные файлы.

**Interaction flow:**
1. Вход: любая сессия, создающая/меняющая данные ученика.
2. Данные пишутся только в игнорируемые git-ом пути.
3. Success: `git status` в форке чист от личных файлов после любой сессии.
4. Failure: личный файл виден в git status → коммит не выполняется, файл добавляется в исключения.

**Exception handling:**
| Scenario | Handling | User-facing message |
|---|---|---|
| Новый тип личного файла не попал в исключения | Остановить коммит, дополнить `.gitignore`, только потом коммитить | «Личный файл X не в .gitignore — исправил, коммит после проверки» |
| Личный файл уже закоммичен в историю форка | Удалить из индекса и истории до push; если уже запушен — переписать историю немедленно | Отчёт владельцу о факте и принятых мерах |

### 3.4 Недельная программа: `/coach-plan`

**Description:** Генерация плана недели (3–4 talk, 1 listen+discuss, 1 write, 1–2 review) на основе слабых мест, SRS-очереди и ретро прошлой недели, с коротким интервью о занятости.
**User story:** As ученик, I want получать готовый план недели, подстроенный под мои слабые места и занятость, so that не решать каждый день «чем заняться» и не бросить через две недели.
**Priority:** P1 — запускается только после 5 проведённых `/talk`-сессий (защита от «залипнуть в инженерку»).

**Business rules:**
1. Источники плана: слабые навыки из mastery, топ паттернов ошибок, объём SRS-очереди, ретро прошлой недели (пропущенное → перенести или облегчить).
2. Перед генерацией — короткое интервью: занятость недели, чего хочется, что бесило на прошлой неделе.
3. Состав недели: 3–4 talk-сессии, 1 связка listen+discuss (материал подбирается по интересам и уровню: подкаст, видео или выступление), 1 write, 1–2 review.
4. Суммарная нагрузка недели ≤ дневная цель минут × доступные дни (из интервью).
5. План показывается таблицей; правки вносятся диалогом; итог записывается в план недели.
6. Каданс синхронизирован с Chaush OS: воскресенье — ретро, понедельник — план.
7. Каждая активность имеет тип (talk, listen, discuss, write, review, vocab) и статус (planned, done, skipped); выполненная активность получает ссылку на сессию.

**Interaction flow:**
1. Ученик входит: `/coach-plan` в понедельник (или по напоминанию коуча).
2. Коуч показывает выводы из данных → 2–3 вопроса интервью → черновик плана таблицей.
3. Правки диалогом → подтверждение → план записан.
4. Success: план на неделю в системе, `/coach-today` начинает его вести.
5. Failure: см. исключения.

**Exception handling:**
| Scenario | Handling | User-facing message |
|---|---|---|
| Первый запуск — нет ретро и мало данных | План строится от профиля и сид-паттернов, помечается «калибровочная неделя» | «Первая неделя — калибровочная, скорректируем по факту» |
| Прошлая неделя выполнена <50% | Облегчить новую неделю (меньше активностей), спросить о причинах в интервью | «На прошлой неделе было тяжело — делаю эту легче» |
| Ученик пропустил планирование в понедельник | План можно создать в любой день; неделя строится от текущего дня | «Строю план на остаток недели» |
| SRS-очередь пуста, ошибок мало | Больше talk-активностей, review сокращается до 1 | — |

### 3.5 Ежедневный вход: `/coach-today` и план дня в приветствии

**Description:** Ежедневная точка входа: терминал открылся → одной строкой видно активность дня; команда `/coach-today` подтверждает и запускает нужный скилл.
**User story:** As ученик, I want открыв терминал сразу видеть «сегодня по плану X — погнали?», so that начинать занятие без силы воли и выбора.
**Priority:** P1 (вместе с 3.4).

**Business rules:**
1. При старте сессии, если существует план недели и на сегодня есть активность, — она выводится одной строкой в приветствии.
2. `/coach-today` читает план и сегодняшний день → предлагает активность → по согласию запускает соответствующий скилл (talk/discuss/write/review), передавая ему параметры из плана (сценарий, тему, материал).
3. Если вчерашняя активность пропущена — мягко предложить нагнать или перепланировать; правило тона: «пропуск — это данные, не провал», никакого чувства вины.
4. Writing не переизобретается: `/coach-today` передаёт существующему скиллу письма тему из плана (темы рабочие: письмо инвестору, апдейт команде, ответ на возражение).
5. Завершённая активность помечается `done` тем скиллом, который её выполнил.

**Interaction flow:**
1. Ученик входит: открыл терминал (приветствие) или `/coach-today`.
2. Коуч: «сегодня по плану X, погнали?» → ученик соглашается / просит замену.
3. Запускается целевой скилл с параметрами из плана.
4. Success: активность выполнена и помечена `done` со ссылкой на сессию.
5. Failure: см. исключения.

**Exception handling:**
| Scenario | Handling | User-facing message |
|---|---|---|
| На сегодня активности нет | Предложить внеплановую talk-сессию или ближайшую невыполненную активность | «Плана на сегодня нет — хочешь свободный talk?» |
| Плана недели нет вообще | Предложить `/coach-plan` (если пройден гейт 5 сессий) или свободную сессию | «Плана недели ещё нет — составим?» |
| Ученик отказывается от активности дня | Предложить замену из плана или свободную сессию; исходная активность остаётся planned | «Ок, что вместо этого?» |
| Пропущено несколько дней подряд | Предложить перепланирование недели вместо догонялок | «Давай пересоберём остаток недели» |

### 3.6 Обсуждение материалов: `/discuss`

**Description:** Голосовое обсуждение listening-материала из плана: вопросы-ориентиры до прослушивания, пересказ-мнение-спор после, новая лексика — в SRS.
**User story:** As ученик, I want обсуждать голосом послушанный материал, so that превращать пассивное слушание в активную речь.
**Priority:** P1 (вместе с 3.4).

**Business rules:**
1. В момент планирования listen-активности ученик получает 3 вопроса-ориентира — на что обратить внимание при прослушивании.
2. После прослушивания `/discuss` ведёт голосовое обсуждение по механике `/talk` (те же правила потока, озвучки и отложенного разбора): пересказ, мнение, спор с тезисами автора.
3. Новая лексика из материала добавляется в SRS-очередь.
4. Разбор после обсуждения — идентичен разбору `/talk` (правило 8 модуля 3.1).
5. Активность discuss связана с родительской listen-активностью; обе помечаются `done` после обсуждения.

**Interaction flow:**
1. Ученик входит: `/coach-today` в день discuss-активности (или напрямую `/discuss`).
2. Коуч сверяется с вопросами-ориентирами → голосовое обсуждение → разбор.
3. Success: разбор проведён, лексика в SRS, активности помечены.
4. Failure: см. исключения.

**Exception handling:**
| Scenario | Handling | User-facing message |
|---|---|---|
| Материал не прослушан к дню обсуждения | Перенести discuss, предложить слот для listen | «Материал ещё не слушал? Перенесём» |
| Материал оказался слишком сложным | Обсудить понятую часть; отметить в ретро для калибровки подбора | «Let's talk about the parts that stuck with you» |
| Ученик не согласен с выбранным материалом | Заменить материал, зафиксировать интересы в профиле | «Подберу другой — что интереснее?» |

### 3.7 Прогресс говорения

**Description:** Витрина прогресса, где главное — минуты говорения в неделю и streak именно по speaking, а не общая статистика упражнений.
**User story:** As ученик, I want видеть минуты говорения и speaking-streak, so that понимать, растёт ли главное — практика речи.
**Priority:** P2 (nice-to-have; штатная статистика fluent работает и без этого).

**Business rules:**
1. Главная витринная метрика — минуты говорения в неделю; она вычисляется из логов сессий типов talk/discuss.
2. Streak считается по speaking-активностям, а не по любым касаниям инструмента.
3. Витрина не добавляет новых источников данных — только новое представление существующих баз.

**Interaction flow:**
1. Ученик входит: команда прогресса или вопрос о статистике.
2. Вывод: минуты/неделю, speaking-streak, динамика топ-ошибок.
3. Success: ученик видит тренд за ≥2 недели.
4. Failure: см. исключения.

**Exception handling:**
| Scenario | Handling | User-facing message |
|---|---|---|
| Данных меньше недели | Показать только текущую неделю без трендов | «Тренды появятся со второй недели» |
| В логах сессии без длительности (partial) | Исключить из минут, показать сноской | «N сессий не учтено (прерваны)» |

## 4. Non-Functional Requirements

Акцент внутреннего инструмента: скорость потока и переносимость данных важнее визуала.

| Category | Requirement | Acceptance criterion |
|---|---|---|
| Латентность | Ход разговора в `/talk`: от конца реплики ученика до начала озвучки ответа | ≤ 5 секунд |
| Доступность | Голосовой контур работает офлайн (без облачных сервисов в критическом пути) | Сессия проходит без сети |
| Переносимость данных | Память ученика — читаемые JSON-файлы; сценарии — markdown-промпты; ничего не привязано к конкретной модели | Данные читаются и мигрируются без инструмента |
| Совместимость с upstream | Новые возможности — новыми файлами; правки файлов fluent — точечные | Синк с upstream остаётся дешёвым (черри-пик раз в месяц) |
| Приватность | Личные данные вне публичного git | `git status` форка чист после любой сессии |
| Стоимость обучения пользователя | Нулевая: инструмент живёт в уже привычной среде (терминал Claude Code), вход — 1 команда | Первая сессия без чтения документации |

## 5. Data Requirements (Analytics Events)

Аналитика внутреннего инструмента = записи в существующие базы fluent + план недели. Отдельная событийная шина не заводится.

| Event | Trigger | Key attributes | Purpose |
|---|---|---|---|
| talk_session_completed | Завершение `/talk` с разбором | длительность (мин), сценарий, число ошибок, число фраз в SRS, фокус | Главная метрика: сессии/неделю, минуты говорения |
| talk_session_partial | Прерывание `/talk` до разбора | длительность, причина (если известна) | Отделять зачётные сессии от прерванных |
| mistake_logged | Разбор зафиксировал реальную ошибку | паттерн, пример, severity (blocking/repeating) | Тренд топ-паттернов вниз |
| srs_phrase_added | Разбор добавил фразу в очередь | фраза, источник (talk/discuss) | Наполнение повторений живым материалом |
| discuss_completed | Завершение `/discuss` | материал, длительность, лексика добавлена | Учёт listen+discuss связки |
| week_planned | Подтверждение плана в `/coach-plan` | неделя, число активностей по типам, плановая нагрузка (мин) | База для completed_pct |
| activity_status_changed | Скилл пометил активность done/skipped | id активности, тип, ссылка на сессию | Дисциплина плана, ретро недели |
| week_retro_recorded | Ретро недели при следующем `/coach-plan` | completed_pct, перенесённые активности, заметки | Адаптация следующей недели |

## 6. Acceptance Criteria

| ID | Scenario | Given | When | Then |
|---|---|---|---|---|
| AC-1 | Полная голосовая сессия | Профиль заполнен, озвучка включена | Ученик проводит `/talk` ≥10 минут и говорит стоп-фразу | Разбор из 4 блоков показан; базы обновлены; транскрипт в `results/`; сессия видна в логе |
| AC-2 | Ноль оценок в потоке | Идёт разговор `/talk` | Ученик делает грамматическую ошибку | Коуч отвечает как собеседник без исправления; ошибка появляется только в разборе |
| AC-3 | Бессвязный ввод не штрафуется | Идёт разговор `/talk` | Распознавание выдало бессвязный фрагмент | Коуч переспрашивает; фрагмент отсутствует в mistakes-db после сессии |
| AC-4 | Текстовый fallback | Озвучка отключена в профиле | Ученик запускает `/talk` | Сессия идёт текстом по тем же правилам, разбор полноценный |
| AC-5 | Прерванная сессия не в зачёте | Идёт разговор `/talk` | Терминал закрыт до разбора | Транскрипт сохранён как partial; базы не изменены; активность плана осталась planned |
| AC-6 | Онбординг персонализации | Чистая установка без профиля | Ученик проходит онбординг | Профиль содержит EN/RU, цели work, блок voice; в банке ошибок 7 сид-паттернов с frequency 0 |
| AC-7 | Сид-паттерны не искажают статистику | Сид-паттерны заведены | Проведено 3 сессии без ошибки данного паттерна | frequency паттерна = 0, паттерн не попадает в фокусы разбора |
| AC-8 | Приватность | Проведена сессия с записью данных | Выполняется `git status` в форке | Личные файлы (данные, транскрипты, бэкапы) не отображаются |
| AC-9 | План недели из данных | Проведено ≥5 talk-сессий, есть mistakes/mastery | Ученик запускает `/coach-plan` | План содержит 3–4 talk, 1 listen+discuss, 1 write, 1–2 review; нагрузка ≤ дневная цель × доступные дни |
| AC-10 | Ретро адаптирует неделю | Прошлая неделя выполнена частично | Запускается следующий `/coach-plan` | Пропущенные активности перенесены или неделя облегчена; completed_pct записан в ретро |
| AC-11 | Вход за один шаг | План недели существует, на сегодня есть активность | Ученик открывает терминал | Приветствие содержит одну строку с активностью дня; `/coach-today` запускает её одним подтверждением |
| AC-12 | Пропуск без вины | Вчерашняя активность пропущена | Ученик запускает `/coach-today` | Предложение нагнать или перепланировать; формулировка нейтральная, активность не помечена как провал |
| AC-13 | Discuss связан с listen | В плане связка listen→discuss, материал прослушан | Ученик проводит `/discuss` | Обсуждение идёт по механике talk; лексика в SRS; обе активности помечены done |
| AC-14 | Латентность хода | Голосовая сессия в штатном режиме | Ученик заканчивает реплику | Озвучка ответа начинается ≤5 секунд |
| AC-15 | Уровень определён разговором | Онбординг завершён, уровень не зафиксирован | Проведена первая `/talk`-сессия | Уровень (или диапазон) записан в профиль; формальный тест не проводился |
| AC-16 | Данные восстановимы | Идёт запись результатов сессии в базы | Механизм персиста стартует | Перед записью создан локальный снапшот баз; при сбое записи данные восстановимы из снапшота |

| Risk | Probability | Impact | Mitigation |
|---|---|---|---|
| Залипание в инженерку вместо говорения | Высокая | Критический | Гейт: Фаза 3 только после 5 живых сессий; метрика успеха — сессии, не фичи |

| Оценки просачиваются в поток разговора | Средняя | Высокий | Жёсткое правило «ноль фидбека до конца разговора» в скилле; проверяется в AC-2 |
